# HW Report Section Notes Draft

本文档用于讨论和调整 HW 报告结构规则，不属于最终 `answer.pdf` 正文。

## 封面与信息块

- 封面信息块保留源题编号、姓名或小组成员、主题、实验环境；目录页单独放在后续页面，不写入封面信息块。
- 可使用头像；单人作业一张头像居中即可，组队作业再扩展为多人。
- 小组作业封面画像参照 `HW/04` 的三列表格排版；图片文件统一使用 `assets/pic_1.jpg`、`assets/pic_2.jpg`、`assets/pic_3.jpg`，下方对应列出三名成员姓名。
- 信息块不要变成文件清单。
- 若写“报告说明”，只说明正文组织方式，不列执行命令。

## 题目陈述

- 正文中只保留中文题目陈述，不保留英文原文。
- 忠实翻译原题，保留 `Problem 1`、`(a)`、`(b)` 等编号。
- 原题有小问时，每一个小问单独成节，不把多个小问合并在一个大 `Problem` 结构下。
- 推荐标题形式为 `Problem 1(a)：小问主题`、`Problem 1(b)：小问主题`。
- 题目陈述不写解法，不提前给结论。
- 图片题需要转写或翻译图中的关键文字、数据表和要求。

## 大问标题

- 用罗马数字一级标题表示大问，例如 `# I. 数值方法误差分析 {-}`、`# II. 统计检验与对象比较 {-}`。
- 大问标题只负责分组，不承载 `待求问题`、`解决方式`、`问题答案`、`分析`。
- 大问下的每个小问使用二级标题，例如 `## Problem 1(a)：公式推导与算法设计`。
- 每个小问内部再使用三级标题展开局部结构。
- 若导出时启用自动章节编号，罗马数字大问标题建议标记为不编号 `{-}`，避免出现重复编号。

## 小问结构

- 每个小问内部使用完整局部结构：`待求问题`、`解决方式`、`问题答案`、`分析`。
- 不采用“先全局列所有题目陈述，再全局列所有解决方案，再全局列所有答案”的结构。
- 不采用“一个 Problem 大章里套所有小问，然后只给一套解决方案/答案/分析”的结构。
- 小问之间可以共用前文定义，但引用时要说清楚，不让读者来回翻找。
- 如果某个 Problem 没有小问，则该 Problem 自身就是一个完整单元，也使用同一套局部结构。
- 可在每个 Problem 标题下列出脚本链接，但不要塞在同一行。推荐写法为：

  ```markdown
  相关脚本：

  - [本地 scripts/problemX.py](../../scripts/problemX.py)
  - [GitHub scripts/problemX.py](https://github.com/<owner>/<repo>/blob/main/XX/scripts/problemX.py)
  ```

  链接只指向脚本，不替代正文解释。

## 解决方案

- 写数学模型、变量定义、核心公式和算法设计。
- 需要推导的地方写出推导，不用一句“可得”带过。
- 使用伪代码、流程图、关键公式或少量关键代码片段。
- 不粘贴完整源码。
- 不写“执行某命令生成文件”这类工作流说明。

## 问题答案

- 直接回答题目要求。
- 因为每个小问已经单独成节，本节只回答当前小问，不再混入其他小问结果。
- 结果表、关键图像应放在答案附近。
- 数值结果需要单位、误差、对比基准或判断标准。
- 不把原始日志或文件列表作为答案主体。

## 讨论和扩展

- 解释结果为什么合理、误差来自哪里、方法有什么局限。
- 可讨论数值稳定性、收敛阶、随机误差、模型假设或检验局限。
- 语气保持正式，不写内部协作过程或“我如何组织报告”。
- 对于统计题，明确结论的适用范围，不把弱检验说成充分证明。

## 图表

- 图表放在首次分析它的段落附近。
- 图题和正文说明要能让读者知道图在证明什么。
- 表格列名应包含单位或统计含义。
- 图例、坐标、标签不能拥挤或被裁切。

## 代码与文件

- 完整代码留在 `scripts/` 或任务根目录，不进入正文。
- 报告里只展示解释问题所必需的算法片段或伪代码。
- PDF 中可保留到 `scripts/` 下脚本的本地链接，也可追加 GitHub `blob/main/...` 外部链接，便于不同阅读环境跳转到对应实现。
- “文件结构与说明”只在确实要说明提交包时短写，不作为默认正文主线。
- `docs/answer/answer.md` 是报告正文；规则、模板和注意事项放在单独文档。
- 原生 LaTeX 示例已经冻结为历史试验，不再作为后续作业报告模板应用；新报告优先维护 Markdown/Pandoc 源文件。
